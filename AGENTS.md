# Repository Guidelines

## Project Structure & Module Organization

This is a Bun workspace generated from Better-T-Stack. Application code lives under `apps/`: `apps/web` is the React Router frontend and `apps/server` is the Hono/tRPC backend. Shared packages live under `packages/`: `api` contains routers and business logic, `auth` owns Better Auth setup, `db` contains Drizzle schema and migrations, `env` centralizes typed environment access, `infra` handles Cloudflare/Alchemy deployment, and `ui` contains shared shadcn/ui components and global styles. Tests are colocated with source as `*.test.ts`.

## Build, Test, and Development Commands

Use Bun 1.3.2 as declared in `package.json`.

- `bun install`: install workspace dependencies.
- `bun run dev`: run all development services.
- `bun run dev:web`: run only the React Router app at `localhost:5173`.
- `bun run dev:server`: run only the API server at `localhost:3000`.
- `bun run build`: build all workspace targets through Vite+.
- `bun run check`: run Vite+ checks plus workspace TypeScript checks.
- `bun run test`: run the Bun test suite.
- `bun run db:push`, `db:generate`, `db:migrate`, `db:studio`: manage Drizzle/Postgres schema and tooling.

## Coding Style & Naming Conventions

Write TypeScript using ES modules, named exports where practical, and existing path aliases such as `@dubbed-i/ui/components/button` and `@/lib/auth-client`. Follow the current formatter output: two-space indentation, double quotes, semicolons, and trailing commas where generated by tooling. React components use PascalCase, hooks/helpers use camelCase, and route files follow React Router conventions in `apps/web/src/routes`.

## Testing Guidelines

Tests use `bun:test` and are colocated with implementation files, for example `packages/api/src/routers/index.test.ts` and `apps/server/src/index.test.ts`. Name test files `*.test.ts`, group behavior with `describe`, and mock external modules with `mock.module` when isolating API or database logic. Run `bun run test` before submitting behavior changes and `bun run check` before larger changes.

## Commit & Pull Request Guidelines

The current history uses short, lowercase commit subjects such as `initial commit` and `first deploy`; keep subjects concise and imperative when possible. Pull requests should describe the user-facing change, list verification commands run, link related issues, and include screenshots or recordings for UI changes. Note any database migrations, new environment variables, or deployment steps explicitly.

## Security & Configuration Tips

Copy `.env.example` files for local development and never commit real `.env` files. Required examples exist in `apps/server`, `apps/web`, and `packages/infra`. Rotate any credential that appears in shared logs, issues, scripts, or commits.
